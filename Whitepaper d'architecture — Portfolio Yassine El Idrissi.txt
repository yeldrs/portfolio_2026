1. Vue d'ensemble
1.1 Ce qu'est le site
Portfolio personnel de designer produit / UX, mono-page à l'accueil avec pages de détail par étude de cas. Trois projets sont publiés (900.care, Caisse des Dépôts, Batchcooking). Le site est 100 % statique : aucun serveur, aucune base de données, aucune logique runtime. Tout est pré-généré au build.

2.2 Le routing
Astro utilise le file-based routing : la structure de src/pages/ définit les URLs.

index.astro → /
about.astro → /about
404.astro → page 404
work/[slug].astro → /work/900care, /work/caissedesdepots, etc.
Les crochets [slug] signalent une route dynamique. Une seule page-gabarit génère autant de pages HTML qu'il y a de projets.

2.3 Le mécanisme central : Content Collections + getStaticPaths
C'est le cœur du site. Trois pièces travaillent ensemble :
1. Le schéma (src/content/config.ts) définit et valide la forme de chaque projet via Zod. Si un fichier .md ne respecte pas le schéma (champ obligatoire manquant, mauvais type), le build échoue. C'est une protection : impossible de publier un projet malformé.
2. Les fichiers de contenu (src/content/projects/*.md) contiennent les données de chaque projet dans leur frontmatter (titre, client, métriques, images…).
3. La génération (work/[slug].astro) : la fonction getStaticPaths() lit tous les projets de la collection et produit une page par projet, en utilisant le champ semanticSlug comme URL.

Fichier .md  →  validé par config.ts  →  lu par getStaticPaths()  →  page HTML générée
2.4 Build et sortie
astro.config.mjs fixe :

output: 'static' → génération 100 % statique
build.format: 'file' → génère /about.html plutôt que /about/index.html
trailingSlash: 'ignore' → tolère les deux formes d'URL
des redirections (/work → /#projects, /aboutme → /about, etc.)
⚠️ Ces redirections sont dupliquées dans astro.config.mjs ET dans .htaccess. Voir §7.

Design System (partie designer)
Cette section est autosuffisante : un designer peut modifier l'apparence du site sans toucher à la logique Astro, en comprenant la chaîne ci-dessous.

4.1 La chaîne token → classe → composant
Le design system fonctionne en trois niveaux, du plus abstrait au plus concret :

tokens.js (primitives)   →   tokens.js (semantic)      →   composant .astro
"#4364e8"                →   'background-brand'        →   class="bg-background-brand"
                             ↑ consommé tel quel par
                             tailwind.config.mjs

Niveau 1 — tokens.js › primitives : les valeurs brutes. Couleurs réelles, tailles de police, espacements. La couleur de marque est la palette accent-action (violet), du 50 (très clair) au 950 (très foncé). Palette de gris + surface.page + base blanc/noir complètent l'ensemble.
Niveau 2 — tokens.js › semantic.colors : donne des noms de RÔLES aux valeurs (background-brand, button-background-primary, text-body-accent…) en RÉFÉRENÇANT directement les primitives (valeurs réelles, pas des placeholders). C'est la couche que le designer manipule le plus.
Niveau 2 bis — tailwind.config.mjs : ne fait plus que CONSOMMER cette couche (colors: { ...tokens.semantic.colors }). Il n'y a donc plus de mapping dupliqué à maintenir, ni de risque de divergence. On n'édite tailwind.config que pour la typo, les espacements, les rayons — pas les couleurs sémantiques.
Niveau 3 — les composants : utilisent les classes sémantiques. Un composant écrit class="text-text-body-accent" sans jamais connaître le code couleur exact.

Pourquoi cette structure ? Changer la couleur de marque partout sur le site = modifier une seule valeur dans tokens.js (primitives), ou re-pointer un rôle dans tokens.js (semantic.colors). Toute la cascade se met à jour, y compris Tailwind. C'est la force du système — et la raison de ne jamais coder une couleur « en dur » dans un composant.
4.2 Les tokens disponibles
Couleurs sémantiques (définies dans tokens.js › semantic.colors, consommées par Tailwind) :

Fonds : background-primary (blanc), background-brand (violet accent-action), background-page (canvas #f4f7f8), background-accent-primary, background-disabled
Textes : text-body-primary (quasi-noir), text-body-secondary (gris), text-body-tertiary (gris clair), text-body-accent (violet), text-heading-accent (violet)
Boutons : button-background-primary + états hover/active, variantes secondary
Bordures : stroke-primary (+ hover/dark), stroke-secondary (gris clair, séparateurs), stroke-neutral-primary, stroke-tertiary (+ hover), stroke-disabled
Typographie : police Manrope (chargée dans BaseLayout), échelle de sm à 6xl avec interlignage pré-calculé. Les titres suivent une convention : H1 = text-4xl, H2 = text-3xl, H3 = text-lg.
Espacements : échelle nommée xxs (2px) → 6xl (112px). Utilisée pour marges, paddings, gaps.
Rayons : md (6px), xl (12px), card (12px).

5. Flux de données : du fichier au rendu
Parcours complet d'un projet, du .md à la page en ligne :
1. Écriture — Un projet est un fichier src/content/projects/{nom}.md. Son frontmatter contient toutes les données (titre, client, contexte, métriques, images…).
2. Les images par index — Le champ projectImages est un tableau où la position compte :

Index 0 → image de contexte (principale)
Index 1 → image de contexte (secondaire, optionnelle)
Index 2 → image de rôle
Index 4 → image de conception
Index 6 → image de résultats
Index 8+ → carrousel
work/[slug].astro lit ces positions exactes pour placer chaque image dans la bonne section.

⚠️ Point fragile majeur. Le mapping repose entièrement sur l'ordre du tableau. Insérer une image au mauvais index décale tout. Laisser une chaîne vide "" pour garder une position sans image. Voir §7.
3. Validation — Au build, config.ts vérifie chaque frontmatter contre le schéma Zod. Erreur = build stoppé.
4. Liste sur l'accueil — index.astro appelle getCollection("projects"), filtre les brouillons (isDraft), trie par date, et affiche une ProjectCard par projet.
5. Page de détail — getStaticPaths() dans [slug].astro génère une page par projet, en utilisant semanticSlug comme URL et en injectant le contenu MDX rendu.
6. Sortie — HTML statique dans dist/, prêt à héberger.
6. Guide de maintenance
6.1 Ajouter un nouveau projet
Créer src/content/projects/{nom}.md.
Copier le frontmatter d'un projet existant comme gabarit (900care.md est le plus complet).
Renseigner tous les champs obligatoires du schéma (title, client, role, roleDescription, context, problem, keyInsights, methodology, delivery, metrics, cardImage, semanticSlug, publishDate).
Déposer les images dans public/images/{projet}/ et les référencer dans le bon ordre dans projectImages (respecter les index de §5).
semanticSlug doit être en kebab-case (minuscules, tirets) — sinon le schéma rejette.
npm run build : si le build passe, le projet apparaît automatiquement sur l'accueil et à /work/{semanticSlug}.
Astuce : mettre isDraft: true pour préparer un projet sans le publier.
6.2 Éditer le design
Voir §4.3. Règle d'or : modifier les tokens, jamais les valeurs en dur dans les composants.

6.3 Modifier une page
Accueil : src/pages/index.astro
À propos : src/pages/about.astro
En-tête / pied commun : Navbar.astro / Footer.astro
Métadonnées SEO globales : BaseLayout.astro
6.4 Lancer le projet en local
npm install
npm run dev      # serveur de dev
npm run build    # build de production dans dist/
npm run preview  # prévisualiser le build
6.5 Hébergement (état actuel) et migration
Réalité du setup : le SITE est servi par GitHub Pages (workflow .github/workflows/deploy.yml + fichier CNAME). Hostinger n'est QUE le registrar du domaine (yassineelidrissi.com) et le fournisseur d'email (MX) — pas l'hébergeur web. Le DNS du domaine pointe vers GitHub Pages.
Conséquence : les redirections/URLs propres actives sont celles d'astro.config.mjs. Le fichier .htaccess (Apache) est ignoré par GitHub Pages — il est conservé comme filet de sécurité dormant au cas où le site migrerait vers un hébergeur Apache, mais il n'a aucun effet aujourd'hui (voir §7.4).
Avant toute migration : adapter les redirections à l'hébergeur cible (astro.config.mjs pour un statique/GitHub Pages ; .htaccess pour Apache/Hostinger), et vérifier que CNAME et le champ site: d'astro.config.mjs pointent vers le même domaine.
7. ⚠️ Warnings — points fragiles et dette technique
Classés par gravité.

🟠 Important — sources de bugs silencieux
7.1 Hébergement clarifié, config double conservée. Le site tourne sur GitHub Pages (deploy.yml + CNAME) ; Hostinger = registrar + email seulement. Le .htaccess (Apache) est donc inactif sur la prod actuelle, mais volontairement conservé comme filet dormant. Point de vigilance : tant que les deux stratégies coexistent, garder à l'esprit que SEUL astro.config.mjs régit réellement les redirections en prod. Voir §6.5 et §7.4.
7.2 Design system consolidé en source unique (résolu). Anciennement : tokens.js › semantic.colors n'était pas lu par Tailwind, qui dupliquait le mapping à la main → divergence (brand turquoise vs violet). Désormais tailwind.config.mjs consomme colors: { ...tokens.semantic.colors } : une seule source, plus de divergence possible. Règle : éditer les couleurs sémantiques UNIQUEMENT dans tokens.js. Voir §4.3.
7.3 Mapping d'images par index positionnel.
Dans les MDX, les images sont placées par leur position dans projectImages (0=contexte, 2=rôle…). Fragile : un décalage casse silencieusement la mise en page, sans erreur de build. Toujours conserver les positions vides avec "". Une évolution souhaitable serait de nommer les images (objet clé/valeur) plutôt que de les indexer.
7.4 Redirections dupliquées. Les mêmes redirections vivent dans astro.config.mjs ET .htaccess. En prod (GitHub Pages) seul astro.config.mjs s'applique ; le .htaccess ne prendrait le relais que sur Apache. Tant que le site est sur GitHub Pages, astro.config.mjs est la source de référence — modifier le .htaccess seul reste sans effet (§6.5).

🟡 À nettoyer — dette sans impact immédiat
7.5 Script de validation non intégré. validate-frontmatter.cjs dépend de gray-matter (absent de package.json) et n'est lié à aucun script npm → actuellement non fonctionnel. Soit le documenter comme outil manuel (npm install gray-matter && node validate-frontmatter.cjs), soit le supprimer — le schéma Zod de config.ts couvre déjà la validation au build.
7.6 Sitemap manuel malgré l'intégration disponible. @astrojs/sitemap est installé (devDependencies) mais absent des integrations d'astro.config.mjs. Le sitemap.xml est donc écrit à la main dans public/ — il faut le mettre à jour manuellement à chaque nouveau projet. Activer l'intégration automatiserait cela.
7.7 Brouillons et artefacts versionnés. src/content/projects/biomimicry.txt (brouillon en .txt non chargé, image absente), repomix-output.xml (~126 Ko régénérable) et la règle morte de global.css (variables CSS --color-bg / --color-text jamais définies) restent à trancher. Voir §3.4.
7.8 Taille de [slug].astro. ~680 lignes dans un seul fichier-gabarit. Fonctionnel mais dense. Si les études de cas se complexifient, envisager d'extraire les sections répétées (Diagnosis, Conception, Results) en sous-composants.
Annexe — Résumé exécutif pour reprise rapide
Nature : portfolio Astro statique, 3 projets, hébergé sur GitHub Pages (domaine + email chez Hostinger).
Pour ajouter un projet : un fichier .md dans content/projects/, respecter le schéma et l'ordre des images. Rien d'autre.
